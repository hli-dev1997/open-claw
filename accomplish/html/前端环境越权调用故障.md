\## 🚨 故障现场深度复盘

\- \[x] \*\*现象还原\*\*：在浏览器访问 `http://127.0.0.1:18789/` 时，白屏且 Console (控制台) 报错 `process.cwd is not a function` 或 `process is not defined`。

\- \[x] \*\*越权本质\*\*：`process.cwd` 是 Node.js 独有的系统底层 API，用于获取当前服务器的工作目录。但 `home-dir.ts` 这个文件被错误地 Bundle (打包) 进了发给浏览器执行的 JS 文件中。浏览器作为 Client (客户端)，出于沙箱安全机制，根本不存在 `process` 这个全局对象。

\- \[x] \*\*旧方案为何失效 (假动作)\*\*：

&#x20; 你在 `index.html` 里写了 `<script> window.process = {...} </script>` 试图打补丁。这就好比在 Java 里用 ThreadLocal (线程局部变量) 塞了个假对象。

&#x20; 但失效原因在于：现代前端采用 ES Module (ECMAScript 模块化标准，即 `<script type="module">`)，它天生具有\*\*独立且封闭的作用域 (Isolated Scope)\*\*。模块内部的代码在严格模式下，不会自动去全局 `window` 对象上寻找未定义的变量，导致运行时依然报错。



\## ⚙️ 核心解法与原理剖析

\- \[x] \*\*精准改刀\*\*：在 `ui/vite.config.ts` (前端构建配置文件) 中，利用 Vite 的 `define` 特性注入静态对象：

&#x20; ```typescript

&#x20; define: {

&#x20;   "process.env": {},

&#x20;   "process.cwd": "(() => '/')",

&#x20;   "process.platform": JSON.stringify("browser"),

&#x20;   // ...

&#x20; }

&#x20; ```

\- \[x] \*\*降维打击原理 (Compile-time vs Runtime)\*\*：

&#x20; 这个解法抛弃了“在浏览器里打补丁”的思路，而是直接在\*\*构建阶段 (Build Phase)\*\* 动手。

&#x20; > \*\*Java 类比\*\*：这就像是 Lombok 的 `@Data` 注解或者 Maven 的资源过滤 (Resource Filtering)。它不是在程序跑起来的时候通过 Reflection (反射) 动态给属性赋值，而是在 Javac (Java 编译器) 编译阶段，直接修改了 Abstract Syntax Tree (AST / 抽象语法树)。

&#x20; 当 Vite 遇到代码里的 `process.cwd()` 时，直接将其在生成的 `.js` 文件里“物理替换”成了 `(() => '/')()`。这样，浏览器拿到的代码里根本连 `process` 这个词都没有，自然完美规避了报错。

\- \[x] \*\*产物刷新\*\*：执行 `pnpm ui:build` 生成新的 dist (分发) 目录，并强制刷新浏览器缓存，确保加载的是经过 AST (抽象语法树) 替换后的安全代码。

\## ❓ 项目追问清单 (可作为八股文复习提纲)

\- \[ ] \*\*架构耦合反思\*\*：既然 `home-dir.ts` 涉及 Node.js 的文件系统操作，为什么前端的 UI 页面会去引用它？这是否说明项目的 Dependency Graph (依赖图谱) 存在前后端代码未严格隔离的坏味道 (Bad Smell)？

\- \[ ] \*\*同构开发的风险\*\*：在 Isomorphic (同构/跨平台) JavaScript 开发中（代码既跑在 Node 又跑在浏览器），除了使用 Vite/Webpack 的 `define` 插件进行 Hardcode (硬编码) 替换，是否还有更优雅的方案？（例如：条件导出 `Conditional Exports` 或在代码里加 `typeof window !== 'undefined'` 判断）。

\- \[ ] \*\*构建管线缺失\*\*：从日志得知，官方在 2026.3.22 版本后移除了内置的前端编译产物。在咱们后续部署量化交易系统时，是否需要在 CI/CD (持续集成/持续部署) 的 Jenkins 或 GitHub Actions 流水线中，显式加上 `pnpm ui:build` 这一步，以防部署到生产环境时依然拉取到纯源码？
