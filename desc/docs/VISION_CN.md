## OpenClaw 愿景

OpenClaw 是真正能做事的 AI。
它运行在你的设备上，使用你已有的频道，遵循你的规则。

本文档阐明了项目的现状和发展方向。
我们仍处于早期阶段，迭代速度很快。
项目概览和开发文档：[`README.md`](README.md)
贡献指南：[`CONTRIBUTING.md`](CONTRIBUTING.md)

OpenClaw 始于一个个人项目，用于学习 AI 和构建真正有用的东西：
一个能在真实计算机上运行真实任务的助手。
经过多次名称和形式的演变：Warelay -> Clawdbot -> Moltbot -> OpenClaw。

目标：打造一个易于使用、支持广泛平台、尊重隐私和安全的个人助手。

当前重点：

优先级：

- 安全和默认安全配置
- Bug 修复和稳定性
- 设置可靠性和首次运行体验

接下来的优先级：

- 支持所有主要模型提供商
- 改进对主要消息渠道的支持（并添加一些高需求渠道）
- 性能和测试基础设施
- 更好的计算机使用和代理框架能力
- CLI 和 Web 前端的人体工程学
- macOS、iOS、Android、Windows 和 Linux 上的伴侣应用

贡献规则：

- 一个 PR = 一个问题/主题。不要打包多个不相关的修复/功能。
- 超过约 5,000 行改动的 PR 仅在特殊情况下进行审查。
- 不要一次打开大批小 PR；每个 PR 都有审查成本。
- 对于非常小的相关修复，鼓励合并为一个有重点的 PR。

## 安全

OpenClaw 中的安全是一种刻意的权衡：强大的默认设置而不牺牲功能。
目标是在进行真实工作时保持强大，同时让危险路径明确且由操作者控制。

规范安全政策和报告：

- [`SECURITY.md`](SECURITY.md)

我们优先考虑安全默认设置，但也为可信的高功率工作流公开清晰的控制。

## 插件和内存

OpenClaw 拥有广泛的插件 API。
核心保持精简；可选功能通常应作为插件提供。

首选插件路径是 npm 包分发加本地扩展加载用于开发。
如果你构建插件，在你自己的仓库中托管和维护它。
向核心添加可选插件的门槛故意设得很高。
插件文档：[`docs/tools/plugin.md`](docs/tools/plugin.md)
社区插件列表 + PR 审查标准：https://docs.openclaw.ai/plugins/community

内存是一个特殊的插件槽，一次只能有一个内存插件处于活动状态。
目前我们提供多个内存选项；随着时间推移，我们计划收敛到一个推荐的默认路径。

### 技能

我们仍然为基线体验提供一些捆绑的技能。
新技能应首先发布到 ClawHub（`clawhub.ai`），而不是默认添加到核心。
核心技能添加应该很少，并需要强大的产品或安全原因。

### MCP 支持

OpenClaw 通过 `mcporter` 支持 MCP：https://github.com/steipete/mcporter

这使 MCP 集成灵活且与核心运行时解耦：

- 无需重启网关即可添加或更改 MCP 服务器
- 保持核心工具/上下文表面精简
- 减少 MCP 变动对核心稳定性和安全性的影响

目前，我们更倾向于这种桥接模式，而不是在核心中构建第一流的 MCP 运行时。
如果 `mcporter` 还不支持某个 MCP 服务器或功能，请在那里提出问题。

### 设置

OpenClaw 目前按设计以终端优先。
这使设置明确：用户提前看到文档、身份验证、权限和安全态势。

从长远来看，随着加固的成熟，我们希望有更容易的入职流程。
我们不希望有隐藏关键安全决策的便利包装。

### 为什么选择 TypeScript？

OpenClaw 主要是一个编排系统：提示、工具、协议和集成。
选择 TypeScript 是为了让 OpenClaw 默认可被破解。
它被广泛使用，快速迭代，易于阅读、修改和扩展。

## 我们暂时不会合并的内容

- 当它们可以存放在 ClawHub 上时的新核心技能
- 所有文档的完整文档翻译集（延迟；我们计划稍后进行 AI 生成翻译）
- 不明确符合模型提供商类别的商业服务集成
- 围绕已支持渠道的包装渠道，没有明确的功能或安全缺口
- 当 `mcporter` 已经提供集成路径时，核心中的第一流 MCP 运行时
- 代理层次框架（管理者之管理者/嵌套规划器树）作为默认架构
- 重量级编排层，重复现有代理和工具基础设施

这个列表是路线图护栏，不是物理定律。
强大的用户需求和强大的技术基本原理可以改变它。
